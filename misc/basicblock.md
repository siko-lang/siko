# Basic Block Lowering Plan

This document captures the staged plan for moving Siko's current post-lowering
HIR, which is still mostly the AST, toward a basic-block-shaped HIR. The goal is
not to introduce a separate IR yet. The goal is to make each function body a
single flat `Block(Vec[Statement])` where labels split the list into implicit
basic blocks and explicit jumps connect those blocks.

The pass is intended to run late, after monomorphization and the existing
language-level lowerings, immediately before backend lowering.

## Target Shape

After basic block lowering, a function body should be one flat block:

```text
{
    entry_label:
    ...
    if cond {
        goto then_label;
    } else {
        goto else_label;
    };

    then_label:
    ...
    goto join_label;

    else_label:
    ...
    goto join_label;

    join_label:
    ...
    return value;
}
```

The "blocks" are implicit:

- A `Label` starts a block.
- A block ends at `Return`, `Goto`, `If`, or `Switch`.
- `loop`, `break`, and `continue` are removed and replaced with labels and
  gotos.
- `Match` is removed before the backend sees the lowered function body.

Calls to functions returning `!` are semantically divergent, but they are not
LLVM terminators. This pass does not need to add an `unreachable` AST node. A
future LLVM backend can emit `unreachable` after calls to `!` functions.

## New AST Primitives

The current AST already has `Label` and `Goto`. The basic block work needs three
additional low-level expression forms.

```text
EnumTag(value: Expr)
Switch(tag: Expr, arms: Vec[Expr])
ReadVariant(value: Expr, tag: Expr)
```

`EnumTag(value)` returns the compiler-owned enum tag. Enum tags are not user
visible. They are guaranteed to start at 0, be dense, and follow
`EnumDef.variants` order.

`Switch(tag, arms)` is an enum-tag dispatch. `arms[index]` is selected when
`tag == index`. This can stay vector-shaped because enum tags are dense and
ordered by definition.

`ReadVariant(value, tag)` extracts the payload struct for the selected variant.
The result type tells us which real payload struct is expected. The `tag`
argument keeps the operation explicit and mirrors the dispatch value, but the
operation is still statically typed by the expression's `ty`.

Example target shape:

```text
let tag = EnumTag(value);
switch tag [
    goto arm_0,
    goto arm_1,
];

arm_0:
let payload = ReadVariant(value, tag);
let v = payload.item0;
...
goto join;

arm_1:
...
goto join;

join:
...
```

## Real Variant Payload Structs

Variant payload structs must be real `Program.structs` entries, not hidden C
backend artifacts.

For an enum variant like:

```text
enum Option[T] {
    Some(T),
    None,
}
```

the compiler should create a real payload struct for `Some`, with normal fields:

```text
struct <payload for Some[T]> {
    item0: T,
}
```

Variants without payload can either have an empty payload struct or avoid
`ReadVariant` in the corresponding arm. The simpler invariant is to create real
payload structs for every variant, including empty ones, because then enum
layout is uniform.

These structs are generated by the HIR lowering pipeline, not by the C backend.
Once generated, the backend lowers them like any other struct.

Field names should be compiler-owned and stable, for example:

```text
item0
item1
...
```

The pass that creates these structs must also make sure enum construction and
enum storage layout agree with them. After this point, the enum container's
payload union should contain these real struct types.

## Backend Boundary

The backend should not know about match patterns or pattern binding.

After the basic block pipeline:

- `Match` should not reach the backend.
- Pattern bindings should already be ordinary `let` statements and field
  accesses.
- Variant payload structs should already exist in `Program.structs`.
- The backend only lowers:
  - normal structs,
  - enum containers,
  - `EnumTag`,
  - `Switch`,
  - `ReadVariant`,
  - labels and gotos.

The backend may still know enum memory layout, because `EnumTag` and
`ReadVariant` are explicit low-level enum operations. It should not synthesize
variant payload structs or create pattern binding locals.

## Staged Implementation

This likely needs several smaller passes/steps.

### 1. Add the New Expression Kinds

Introduce the AST expression variants first without changing the pipeline:

```text
ExprKind.EnumTag(Expr)
ExprKind.Switch(Expr, Vec[Expr])
ExprKind.ReadVariant(Expr, Expr)
```

Required compile-only plumbing:

- Add formatter output for the new expressions.
- Add sanity checker cases, initially accepting them or checking their children.
- Add C backend cases if required by exhaustiveness, but they can panic until
  the expressions are actually produced.
- Update auto-map/auto-fold style passes where exhaustive matching requires it.

Most earlier passes may panic if these expressions appear, because they should
not appear before late lowering.

The goal of this phase is only: the compiler still compiles and existing tests
continue to pass.

### 2. Make `LowerMatch` Leave Only Enum Matches

Before a basic-block match lowering pass exists, validate what reaches the late
pipeline.

Expected invariant:

- Struct matching is lowered before basic block lowering.
- Int matching is lowered before basic block lowering.
- Char matching is lowered before basic block lowering.
- String matching, if still represented as `Match`, should also be lowered or
  explicitly ruled out.
- Only enum matches can reach the later basic-block match destruction.

This likely belongs in the existing `LowerMatch` pass. It already compiles
complex patterns into simpler control flow. The next step is to validate that
non-enum matches are converted into `if` chains or ordinary destructuring before
the new basic-block pass runs.

Once validated, add assertions in the late lowering path so a non-enum `Match`
fails loudly.

### 3. Generate Real Variant Payload Structs

Add a lowering step that creates real payload structs for enum variants and
updates enum layout expectations accordingly.

Open design points:

- The generated qualified name scheme should be deterministic and based on the
  monomorphic variant qualified name.
- Field names should be compiler-owned: `item0`, `item1`, etc.
- Payload structs should inherit value-type behavior consistently with the enum
  representation.
- Constructors must build enum values using these payload structs.

This step should happen before backend lowering and before `ReadVariant` is
lowered by the backend.

### 4. Implement `LowerBasicBlock`

Create `siko/LowerBasicBlock.sk`.

This pass receives a `Program` and mutates or returns a `Program` whose function
bodies are flat basic-block-style AST blocks.

Responsibilities:

- Flatten nested blocks into the function-level statement list.
- Introduce labels.
- Convert `loop` into labels and gotos.
- Convert `break` and `continue` using a loop-label stack.
- Convert `if` into an `If` terminator whose branches only `goto` labels.
- Convert enum `match` into:
  - `EnumTag`,
  - `Switch`,
  - labelled arm blocks,
  - `ReadVariant`,
  - ordinary field accesses for bindings,
  - gotos to a join label where needed.

The pass should preserve existing normalized assumptions:

- Control-flow expressions are already in statement position.
- Values needed across branches have already been assigned to temps by the
  normalizer.
- Function bodies usually already end in explicit `return`, except for
  semantically divergent bodies.

### 5. Wire the Pass Before Backend Lowering

Insert the pass immediately before the C backend:

```text
value-types
lower-basic-block
c
```

This lets value-type calculation keep operating on the current HIR shape and
lets backends consume the lower-level block-shaped HIR.

### 6. Teach C Backend the New Low-Level Primitives

Once `LowerBasicBlock` produces the new expression forms, the C backend should
lower them directly:

- `EnumTag(value)` reads the enum tag field.
- `Switch(tag, arms)` emits a C `switch`.
- `ReadVariant(value, tag)` extracts the real payload struct for the result
  type.

The backend should no longer lower high-level `Match` after this point. Seeing a
`Match` in the C backend should become an error once the new pass is active.

## Example: Enum Match

Source-level shape:

```text
match opt {
    Some(v) => use(v),
    None => fallback(),
}
```

Late basic-block HIR:

```text
entry:
let tag = EnumTag(opt);
Switch(tag, [
    goto match_arm_0,
    goto match_arm_1,
]);

match_arm_0:
let payload = ReadVariant(opt, tag);
let v = payload.item0;
let tmp = use(v);
goto match_join;

match_arm_1:
let tmp = fallback();
goto match_join;

match_join:
...
```

If an arm diverges, it does not need to jump to the join label:

```text
match_arm_0:
return value;

match_arm_1:
...
goto match_join;
```

## Example: Loop

Before:

```text
loop {
    body;
    if cond {
        break;
    } else {
        continue;
    }
}
after;
```

After:

```text
loop_start:
body;
if cond {
    goto loop_break;
} else {
    goto loop_start;
};

loop_break:
after;
```

## Invariants to Check

After `LowerBasicBlock`:

- Function body is `ExprKind.Block`.
- No `Statement.Loop`.
- No `Statement.ForLoop`.
- No `Statement.WhileLoop`.
- No `ExprKind.Break`.
- No `ExprKind.Continue`.
- No `ExprKind.Match`.
- No `ExprKind.IfLet`.
- Each implicit basic block starts with a `Label`.
- Each implicit basic block ends with `Return`, `Goto`, `If`, or `Switch`, except
  blocks ending in a call to a `!` function if we intentionally allow that until
  LLVM lowering.

## Why This Direction

The earlier idea of keeping `Match` and making arms jump to real labelled bodies
made pattern bindings awkward. C switch cases would have created locals scoped
inside the case, while the real arm body lived after the switch.

Lowering enum matches into `EnumTag`, `Switch`, `ReadVariant`, and real payload
struct field accesses avoids that problem:

- Pattern binding becomes normal code in the selected labelled block.
- C backend no longer performs match-specific binding work.
- LLVM lowering later gets an explicit CFG and simple enum primitives.
- Variant payload representation becomes a real part of `Program`, not backend
magic.
